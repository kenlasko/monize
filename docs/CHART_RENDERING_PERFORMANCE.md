# Chart Rendering Performance: Balance History vs. Portfolio Value

This document provides a technical comparison of the performance characteristics of two primary charts in Monize: the **Balance History Chart** (Transactions page) and the **Portfolio Value Over Time Chart** (Investments page). It explains why they performed differently and details the optimization strategies implemented.

---

## Architectural Comparison

| Feature | Balance History Chart (`/accounts/daily-balances`) | Portfolio Value Over Time Chart (`/net-worth`) |
| :--- | :--- | :--- |
| **Data Source** | Raw `transactions` ledger history. | Pre-calculated historical snapshots (`net-worth` tables). |
| **Calculation Method** | **On-the-fly CTE**: Computes daily balances by performing a cumulative sum window function starting from opening balances. | **Pre-calculated Lookup**: Selects static snapshot values already computed and cached in the database. |
| **Resolution Strategy** | **Daily**: Originally returned 1 data point per day, regardless of date range. | **Adaptive**: Automatically switches to monthly resolution for ranges > 2 years. |
| **Render Load (SVG)** | Extremely high for multi-year queries (thousands of daily points per account). | Low (switches from daily to monthly aggregates for large ranges). |

---

## Technical Details

### 1. Cumulative Calculation Cost (Balance History)
The Balance History query executes a cumulative window function over the daily sequence generated by `generate_series`:
```sql
COALESCE(SUM(dtx.total) OVER (
  PARTITION BY ta.id ORDER BY d.dt
  ROWS UNBOUNDED PRECEDING
), 0)
```
For a 10-year period, this computes 3,650 days per account. If a user has 5 accounts, that equals 18,250 points. Serializing, transferring, and parsing this JSON payload on the client, and subsequently rendering an SVG path in Recharts with thousands of points, caused noticeable UI lag.

### 2. Adaptive Resolution (Portfolio Value)
The Portfolio Value chart component (`InvestmentValueChart.tsx`) dynamically adapts its dataset based on the range:
```typescript
const DAILY_RANGES = new Set(['1w', 'mtd', '1m', '3m', 'ytd', '1y', '2y']);
const useDaily = !isIntraday && DAILY_RANGES.has(dateRange);

if (useDaily || isIntraday) {
  // Queries daily snapshot values
  const data = await netWorthApi.getInvestmentsDaily(params);
} else {
  // Switches to monthly summaries for long ranges (5Y, All)
  const data = await netWorthApi.getInvestmentsMonthly(params);
}
```

## Optimization Implemented for Balance History

To replicate the performance characteristics of the Portfolio Value chart, we implemented the **Fast Render** optimization for the Balance History chart:

1. **Keep Calculations Correct**: The PostgreSQL CTE still calculates running sums daily to maintain 100% accurate balances.
2. **Dynamic Query Downsampling**: If `optimize` is set to `true`, the outer select filters the results dynamically in the backend:
   - **Range > 3 Years**: Returns only the 1st of each month and the final end date (`EXTRACT(DAY FROM date) = 1 OR date = $4::DATE`).
   - **Range > 1 Year**: Returns only Mondays and the final end date (`EXTRACT(ISODOW FROM date) = 1 OR date = $4::DATE`).
   - **Range <= 1 Year**: Returns daily values.
3. **UI Control**: Added a **Fast Render** slide toggle switch next to the date range selectors in [BalanceHistoryChart.tsx](file:///h:/AiModelLearning/QuickenReplacements/monize/frontend/src/components/transactions/BalanceHistoryChart.tsx) (persisted in local storage).
